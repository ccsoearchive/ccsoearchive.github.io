I've never used this feature but took a moment to peek at the code on the PTR.

The conversations feature appears removed from the PTR with no apparent replacement.

As Rygarius mentioned in the first post, the C-functions (like BNCreateConversation) are gone, and the default Lua/XML implementation of conversations (BNConversations.xml and BNConversations.lua) are gone.

There is no longer a "Create Conversation" menu item on a bnet friend. I'm not seeing anything in the default code to imply the feature exists somewhere else.

I've not used the conversation feature so I don't know all that was involved, but from what I'm seeing an addon can recreate the conversations feature (once they get BNSendGameData working) but it won't be with the old conversation API to support it. If three or more people are using the addon, the one to initiate the conversation (and those who invite others if they add the ability) can listen for/redistribute messages sent by those he invited. It gets a little complicated if you have more than three in a conversation and the creator isn't a bnet friend with someone who was invited by another, but it's possible to work around.

edit: After giving it a little more thought, the addon wouldn't be overly complicated to do, especially if it's limited to the creator being the only one to invite others to the conversation.

If conversations are meant to be up for a limited amount of time, and only the one creating the conversation is inviting others, and everyone is okay with the conversation ending when the creator leaves, it's very trivial. Messages go to the creator and get resent to the people he invited. Done and done.

Going further, it's still possible if the creator invites others and they invite others too (the creator is not guaranteed to be a bnet friend with whoever their invitee invites):
- When an inviter sends a message it goes to everyone they invited.
- When an invitee sends a message it goes to the inviter (and their invitees if they have any).
- All inviters will be listening for messages from their invitees and resend them to every other of their invitees except the invitee who sent the message.

This should work seamlessly with separate inviters and their own invitees (separate bnet friend group). An invitee that's also an inviter would pass along a message it received from their invitee to their inviter which would get distributed to the parent inviter's invitees.

Management of who is actually in the conversation (so everyone knows who is in the conversation) would work the same way, with tags so you can tell who invited whom to mark on the list. So anyone in the conversation can pull up a list of who's there without needing to be bnet friends with anyone they weren't a friend with before the conversation started.

That said, if someone who invites others leaves or is kicked from the conversation, their invitees would be kicked too because the message link is gone. Ways around this can involve sending/accepting automatic friend requests. For an invitee inviter (an inviter who didn't create the conversation), their orphan invitees can send a bnet friend request up the chain and they become an invitee to the parent inviter. For the conversation creator leaving it's a bit tricker (even without others having the ability to invite). You can have their first invitee become the new "creator" inviter and they would send friend requests to the original creator's invitee list. This is all assuming people are okay with becoming automatic bnet friends with an occasional person in the conversation group--it's definitely not something I'd accept.

So I'd be more inclined to the first simpler option where only the creator can invite and the conversation closes when they leave. No new bnet friend requests need to go out. But I've never used conversations so i don't know how they're commonly used.

03/06/2016 07:34 PMPosted by Galvin
Is there a way to whisper your self in battlenet to test this? or do I have to hunt someone down

There's no way to whisper yourself over battle.net. A second account (or someone else) is required. The next time WoW is on sale for $5 it's a good deal for testing purposes. You can RAF a mount or pet too, and pay for the second subscription entirely with tokens.

That said these changes aren't that drastic. When I wrote a new version of your code the parameter count was throwing me off so I redid it as a vararg (...). After looking at what they've done in more detail, you can probably get by with only changing:
local _, ToonName, Client, RealmName = BNGetToonInfo(ToonID)
to this:
local _, ToonName, Client, RealmName = BNGetGameAccountInfo(ToonID)
And that's it.

The API is relatively unchanged. Here's BNGetFriendInfo from 6.2.3:
presenceID, presenceName, battleTag, isBattleTagPresence, toonName, toonID, client, isOnline, lastOnline, isAFK, isDND, messageText, noteText, isRIDFriend, messageTime, canSoR, isReferAFriend, canSummonFriend = BNGetFriendInfo(index)
and 6.2.4:
bnetIDAccount, accountName, battleTag, isBattleTag, characterName, bnetIDGameAccount, client, isOnline, lastOnline, isAFK, isDND, messageText, noteText, isRIDFriend, messageTime, canSoR, isReferAFriend, canSummonFriend = BNGetFriendInfo(index)
(BNGetFriendInfoByID has the same returns it just uses the bnetIDAccount as a parameter.)

BNGetToonInfo in 6.2.3:
hasFocus, toonName, client, realmName, realmID, faction, race, class, guild, zoneName, level, gameText, broadcastText, broadcastTime, canSoR, toonID = BNGetToonInfo(presenceID or toonID)

BNGetGameAccountInfo in 6.2.4 (some stuff inferred, this may not be 100% accurate):
hasFocus, characterName, client, realmName, realmID, faction, race, class, guild, zoneName, level, gameText, broadcastText, broadcastTime, canSoR, bnetIDGameAccount, bnetIDAccount, isGameAFK, isGameBusy = BNGetGameAccountInfo(bnetIDGameAccount)

The event parameters for CHAT_MSG_BN_WHISPER look the same.

BNSendWhisper will always use the bnetIDAccount.
BNSendGameData will always use the bnetIDGameAccount.
